# GoFurry Ops 未来扩展与演进路线图

## 当前基线

GoFurry Ops 当前已经形成轻量可用的首版闭环：

- `gofurry-ops-agent` 负责节点侧采集、健康检查、HMAC 上报和短暂离线时的 JSONL spool。
- `gofurry-ops-center` 负责接收采样、写入 PostgreSQL、维护服务状态与告警状态，并提供内置监控控制台。
- 控制台已经覆盖总览、节点详情、服务健康、告警、Peer、同步事件和部署事件。
- 部署资产已经具备 systemd、Docker Compose、Nginx 示例和本地测试配置模板。

这份路线图用于指导后续演进。目标不是把 Ops 做成重型平台，而是让它持续保持轻量、可靠、清晰，适合独立维护者长期使用。

## 演进原则

- 轻量优先：默认单 Center、单 PostgreSQL、Agent 低资源占用，不强依赖复杂基础设施。
- 可维护优先：新增能力要有清晰边界、配置示例和验证路径，避免隐式行为堆叠。
- 可扩展但不过度抽象：只在 collector、notifier、事件接入等明显扩展点建立接口。
- 低侵入采集：Agent 默认只做只读检查，不修改业务数据库、Redis 或宿主服务状态。
- 安全默认：生产配置拒绝弱密钥；所有写入入口保留认证、签名、限速和边界校验。
- 单人可运维：部署、备份、升级、故障恢复都应能用少量命令完成。

## 近期阶段：补齐日常运维闭环

目标：让当前首版从“能看见状态”进一步变成“异常能被及时发现、定位和处理”的轻量监控工具。

关键方向：

- 通知渠道：新增 notifier 扩展点，优先支持邮件、Webhook、Telegram 或企业微信中的一到两种；通知内容保持短小，包含节点、服务、级别、最近错误和控制台链接。
- 告警规则配置化：把内存、磁盘、HTTP 连续失败、证书过期、节点离线、同步陈旧等阈值整理成更清晰的配置结构，并在示例配置中标注推荐值。
- 告警抑制与恢复：增加静默窗口、重复通知间隔和恢复通知，避免短时间内重复打扰。
- 面板信息密度：继续打磨总览与节点详情，让最新值、趋势、异常服务、错误消息和采样新鲜度更容易一扫而过。
- 采集配置体验：补充更多配置样例，例如只采系统、只采服务、只采证书、低频采集等常见模式。
- 部署与备份文档：补齐 PostgreSQL schema 备份、配置备份、二进制升级、systemd 回滚和日志排查流程。

价值：

- 异常从“需要打开面板才知道”变成“关键问题会主动提醒”。
- 告警更少误报，恢复路径更清楚。
- 本地测试、生产部署和日常升级都有可照着走的文档。

## 中期阶段：提升扩展性和分析能力

目标：在不变重的前提下，让 Ops 更适合多服务、多节点和多区域的长期运行。

关键方向：

- Agent collector 插件化：把 system、Docker、HTTP、PostgreSQL、Redis、TLS cert 等 collector 的注册、配置和结果模型梳理成稳定扩展点，便于后续增加进程、端口、日志摘要或自定义脚本检查。
- 服务分组与标签：支持给节点和检查项增加 role、env、service、region 等标签，在控制台中按业务服务、环境或区域筛选。
- 事件审计：统一记录管理员登录、配置加载、Agent 上报异常、同步失败、部署事件和告警状态变化，方便回溯。
- 报表与趋势：增加 7 天 / 30 天摘要，展示可用率、资源高水位、告警次数、最慢服务和长期容量趋势。
- Peer 多区域体验：把 Peer 摘要从单纯状态卡片扩展为区域级健康视图，展示跨区域同步状态、告警分布和最后心跳。
- 数据保留策略：将 raw sample、告警事件、部署事件和聚合趋势的保留周期分层配置，控制数据库体积。

价值：

- 新增采集能力时不需要大改 Agent 主流程。
- 多节点状态可以按业务上下文组织，而不是只看机器列表。
- 长期趋势能支持容量判断和故障复盘。

## 长期阶段：形成轻量运维中枢

目标：让 Ops 在保持单体、低依赖的同时，具备跨区域协同、自动化诊断和更成熟的权限边界。

关键方向：

- 多区域联邦：支持多个 Center 互相汇总关键健康信息，单一区域故障时仍可从其它区域看到摘要。
- 自动化 runbook：为常见问题提供可配置的处理建议，例如证书即将过期、节点长时间离线、同步连续失败、数据库延迟异常。
- SLO 与可用性分析：围绕 HTTP、数据库、Redis、同步任务等服务检查计算轻量 SLO 指标，帮助判断服务质量而不仅是瞬时状态。
- 权限体系：从单 passcode 登录演进到多用户、只读/管理员角色、会话审计和可选外部认证。
- 扩展 SDK：为自定义事件、采集器、通知器提供小而稳定的 Go 接口和示例，方便后续在不侵入核心代码的情况下扩展。
- 升级与迁移体系：引入更明确的 schema migration 版本、兼容性说明和升级前检查，降低长期维护成本。

价值：

- GoFurry Ops 可以承担“轻量运维中枢”的角色，而不是只作为临时监控面板。
- 多区域、多人协作和长期数据沉淀有更清楚的边界。
- 扩展能力可控，不需要把项目推向重型平台。

## 暂不做清单

为保持项目克制，以下方向暂不作为默认路线：

- Kubernetes 原生平台化：不默认引入 Operator、Helm、Service Mesh 或复杂集群依赖。
- 重型 APM：不替代 tracing、profiling、日志检索或完整指标平台。
- 复杂 CMDB：不建设资产全生命周期管理，只保留运维所需的节点和服务元数据。
- 强依赖外部消息队列：默认仍以 HTTP 上报和 PostgreSQL 入库为主，除非后续规模确实需要。
- 全自动修复生产故障：runbook 先提供诊断和建议，不默认执行有风险的修复动作。

## 实施节奏建议

- 每次只推进一个明确主题，例如 notifier、标签体系、报表趋势或 Peer 视图。
- 每个主题都应同时包含后端接口、前端展示、配置示例、README 更新和最小测试。
- 对数据库结构有影响的能力优先补 migration 方案，再落业务代码。
- 新增采集器默认关闭，并在示例配置中说明资源消耗与安全边界。
- 保持本地 smoke 流程可运行：Agent once 上报、Center 入库、控制台展示、告警状态变更。
