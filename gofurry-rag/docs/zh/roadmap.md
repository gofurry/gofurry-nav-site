# gofurry-rag Roadmap

这份路线图以当前已经实现的 `gofurry-rag` 为基线，重点不是堆功能，而是把“入库、切分、向量化、检索、回答、观测、运维”这条链路逐步做稳。

## 2026-05 当前状态

当前版本已经不是“只返回 sources 的检索接口”，而是一个可用于内部调试的 RAG 控制台：

- 文档入库、重建、失败重试已经可用
- Chunk 预览、编辑、删除与重新向量化已经可用
- `POST /api/v1/chat/query` 已支持非流式 AI 问答
- `POST /api/v1/chat/stream` 已支持 SSE 流式回答
- 回答默认包含答案、`sources` 与 usage
- `include_details` 可返回完整引用详情，但现在只允许管理员使用
- 腾讯云负责回答生成，本地 Ollama 负责 embedding
- Ollama 已增加并发保护、优先级队列和队列状态观测
- `/api/v1/health` 已改为管理员可见的深度诊断接口

## v0.1.x 已完成

目标：先把最小可用闭环搭起来。

已完成：

- 文本文档和文件导入
- PostgreSQL + pgvector 存储
- 基础检索能力
- 控制台登录态
- 文档列表、Chunk 列表与操作
- 基础 health / overview / worker 状态展示

## v0.2.0 已完成

目标：增强检索调试与评估能力。

已完成：

- 查询结果展示 rank、score、source、chunk 等调试字段
- Chunk 切分预览
- embedding 输入模板增强
- 控制台侧的检索调试体验完善

## v0.3.0 已完成

目标：让知识库具备可重复运维能力。

已完成：

- 批量 reindex
- failed retry
- 文档级 metadata filter
- 控制台按状态、分类、语言、来源等过滤

## v0.4.0 已完成

目标：增强运行安全性、限制和可观测性。

已完成：

- query / ingest timeout 与长度限制
- worker 状态、最近失败、耗时统计
- 部署与回滚文档
- 控制台 observability 面板

## v0.5.0 已完成

目标：把检索结果组织成可调试、可引用的 AI 问答。

已完成：

- 腾讯云 `deepseek-v4-flash` 回答链路
- 基于 sources 的回答生成
- SSE 流式输出
- “答案 + 引用”结构化返回
- 可选引用详情
- 控制台 AI 问答页
- 复制回答 / 复制引用
- 更接近知识库调试台的引用详情面板

## 业务前端接入前还要继续收口的点

虽然当前版本已经适合内部调试，但要直接接业务前端，仍然建议继续补强这些边界：

- 公开问答接口的业务鉴权
- 多租户或知识域范围隔离
- 业务侧引用字段裁剪
- 更细的限流与配额策略
- 面向业务的错误码和降级策略

## 下一阶段建议

优先级建议如下：

1. 把公开问答 API 和业务前端的身份边界彻底定下来
2. 补多轮上下文和会话管理，但继续保持引用可解释
3. 补业务前端可消费的精简 citation 结构
4. 建立真实问题评测集和回归评估
5. 再考虑 rerank、query rewrite、复杂文件解析

## 原则

最重要的一条仍然不变：

```text
先让它找得准、找得稳、找得到依据，再让它回答得更像聊天产品。
```
