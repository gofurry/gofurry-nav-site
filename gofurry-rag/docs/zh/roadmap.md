# gofurry-rag Roadmap

这份路线图已经收束到 v1.0。v1.0 之后暂不继续追加功能规划，生产前重点转为稳定性、配置、备份、监控和小流量验证。

## 当前完成程度

结论：`gofurry-rag` 的核心功能已经完成到 v1.0，可以进入生产更新前检查阶段。

已完成：

- v0.6：公开问答边界收口
- v0.7：前台引用结构精简
- v0.8：站内内容自动同步入库
- v0.9：站内问答场景化入口
- v1.0：轻量多轮上下文

## v0.6：公开问答边界收口（已完成）

已实现公开访客与管理员控制台的边界分离：

- 公开 `/api/v1/chat/query` 和 `/api/v1/chat/stream` 启用更保守的限制
- 公开问题长度、`top_k`、上下文轮数和上下文总长度都有上限
- 公开请求按 IP 做固定窗口限流
- 管理员 Cookie 有效时保留完整调试能力
- 公开响应只返回前台需要的精简来源字段
- 前台对公开错误、限流和模型服务异常有更友好的展示

## v0.7：前台引用结构精简（已完成）

已把公开问答引用收束为面向用户的 citation 体验：

- `gofurry-rag` 公开接口继续输出精简 `sources`
- `gofurry-nav-web` BFF 把公开 SSE 的 `sources` 改写为 `citations`
- `/archive` 内部状态、本地历史和 UI 已切换到 `citations`
- 引用区展示标题、类型、分数、摘要、可选 chunk index 和来源链接
- 复制回答与复制回答加引用分离
- 前台不再依赖 `document_id`、`chunk_id`、`content` 等调试字段

## v0.8：站内内容自动同步入库（已完成）

已完成同步框架和首批高价值站内内容入库：

- 支持控制台手动同步和可选服务内定时同步
- 同步过程基于 checksum 做增量 upsert
- 文档更新时会清理旧 chunks 并重新进入向量化
- 控制台 Sync 页展示同步源、运行状态、预计内容数量和最近结果
- 已接入导航站点、游戏详情、游戏新闻
- 同步源继续共用公开引用裁剪策略，前台协议无需额外扩展

## v0.9：站内问答场景化（已完成）

已把 RAG 能力放到更自然的站内入口：

- `/archive` 保留为自由问答页
- `games` 页右侧工具栏增加游戏问答入口和预设问题
- `nav` 页右侧工具栏承接搜索入口，并增加导航问答入口和预设问题
- 场景化入口统一跳转到 `/archive` 并预填问题
- 回答正文和引用摘要支持 Markdown 渲染

## v1.0：轻量多轮上下文（已完成）

已把 `/archive` 从一问一答历史升级为浏览器本地会话：

- 左侧历史为本地会话列表
- 每个会话最多 20 轮，最多保存 50 个会话
- 会话只保存在浏览器 `localStorage`
- 每次请求携带最近 3 轮已完成问答作为 `context`
- 后端公开 limits 返回上下文轮数和长度限制
- 超出公开上下文限制会返回 `400`
- 检索输入使用当前问题和最近上下文改善追问召回
- Prompt 明确历史回答只用于理解追问、代词和比较对象，不能作为事实依据
- 本轮事实依据只来自本轮检索资料

## 生产前关注点

v1.0 之后暂不继续推进新功能。上线前只做必要的稳态工作：

- 生产配置检查，尤其是认证密钥、Cookie 安全属性、CORS、上游模型地址
- 数据库备份与恢复演练
- RAG 同步任务的小流量验证
- 模型服务不可用、向量化失败、同步失败的告警和日志确认
- 公开问答入口的限流、超时和错误文案验证
- 根据 `docs/code-audit.md` 处理生产前审计发现
