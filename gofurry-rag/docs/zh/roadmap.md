# gofurry-rag Roadmap

这份路线图以当前已经打通的 `gofurry-rag` 和 `gofurry-nav-web` 前后端链路为基线。

后续目标不是把它做成通用 RAG 平台，而是把它演进成 gofurry 个人站里的轻量“内容理解层”：帮访客找东西、解释站内内容、串联导航与游戏资料，同时保持一个人也能长期维护。

## 当前定位

`gofurry-rag` 已经具备完整的基础链路：

- 文档入库、切分、向量化、重建与失败重试
- PostgreSQL + pgvector 知识库存储
- 本地 Ollama embedding 与并发保护
- 腾讯云推理服务回答生成
- 非流式与 SSE 流式问答接口
- 管理控制台、检索调试、引用详情与健康诊断
- `gofurry-nav-web` 业务前端代理接入

下一阶段更适合继续增强“站内知识服务”能力，而不是堆多租户、复杂权限、独立向量服务、Agent 工作流等重功能。

## v0.6：公开问答边界收口（已完成）

优先把业务前端接入后的公共边界做稳。

建议实现：

- 公开问答接口增加轻量 IP 限流
- 明确问题长度、`top_k`、超时等公开参数上限
- 公开接口只返回前台需要的精简字段
- 模型服务异常时返回可读的降级结果
- `gofurry-nav-web` 侧展示更友好的错误与重试状态

目标是让公开入口“可以放心挂在站上”，即使被误用或模型服务短暂不可用，也不会拖垮站点体验。

## v0.7：前台引用结构精简（已完成）

管理台继续保留完整引用详情，业务前台只消费更清爽的 citation 结构。

前台 citation 可以控制在这些字段内：

```text
title
url
source_type
snippet
score
chunk_index
```

建议实现：

- 新增面向业务前端的引用 DTO
- 裁剪调试字段，避免暴露 document/chunk 内部细节
- 回答下方展示“参考来源”
- 来源可点击跳回站内页面或原始链接
- 复制回答时可选择是否附带来源

这个阶段的收益很高：用户能知道回答依据来自哪里，但前台不会变成调试台。

## v0.8：站内内容自动同步入库

让 RAG 真正理解 gofurry 自己的内容，而不是只依赖手动文本导入。

优先入库这些轻量、结构化、维护成本低的数据：

- 导航站点名称、简介、分类、标签、链接
- 游戏条目、介绍、作者、平台、更新时间
- 关于页、更新日志、公告
- 站点维护说明、FAQ、常见问题

建议实现：

- 为导航和游戏数据定义固定的 `source_type`
- 同步任务按 checksum 判断是否需要更新
- metadata 中保留分类、语言、站内路径等过滤字段
- 控制台能按来源类型筛选和重建

目标是让 `gofurry-rag` 变成“懂 gofurry 的搜索和解说层”。

## v0.9：站内问答场景化

把 AI 能力放到用户真正会遇到问题的位置，而不是只放一个独立聊天页。

可以逐步接入：

- `/archive` 保留为自由问答页
- 站点详情页增加“问问这个站点”
- 游戏详情页增加“总结这个游戏”或“找相似作品”
- 导航搜索无结果时提示“用 AI 试试”
- 关于页增加站点说明类问答入口

这些入口要保持克制，优先服务“找资料”和“理解内容”，不要把页面变成泛聊天产品。

## v1.0：轻量多轮上下文

多轮能力可以做，但不需要一开始就做完整会话系统。

建议实现：

- 前端携带最近 2-3 轮问答作为上下文
- 后端仍然以当前问题为主进行检索
- Prompt 中明确上下文只用于消歧，不作为事实来源
- 回答仍然必须基于检索 sources
- 本地浏览器历史继续作为第一阶段存储方案

这样可以支持追问，比如“还有吗”“这个站点呢”“它和上一个有什么区别”，同时不会引入复杂的会话、账号和存储维护成本。

## v1.1：小型真实问题评测集

在继续调参前，先建立一组个人站真实问题。

建议准备 20-50 个问题，例如：

- gofurry 是做什么的？
- 有哪些 furry 相关导航站？
- 某个游戏支持哪些平台？
- 有没有类似某个作品的游戏？
- 最近更新了哪些内容？
- 某类站点在哪里可以找到？

评测重点：

- 是否能召回正确来源
- 回答是否基于来源
- 无资料时是否拒答或说明不足
- 改 chunk、embedding、top_k、prompt 后是否退化

这个比过早引入复杂 rerank 更适合个人维护，也更容易长期坚持。

## v1.2：混合检索

如果后续发现站名、游戏名、作者名等专有名词召回不稳定，再考虑混合检索。

建议路径：

- 保留 pgvector 语义检索
- 增加 PostgreSQL 全文搜索或 trigram 匹配
- 对专有名词命中结果做简单加权
- 控制台展示向量分和关键词分，方便调试

不建议过早引入 Elasticsearch 或独立检索集群。以个人站规模，PostgreSQL 足够支撑很长时间。

## 暂不优先

这些能力可以先不做，避免维护成本失控：

- 多租户
- 复杂账号权限系统
- 大规模网页爬虫
- PDF、DOCX、OCR 全套解析
- Agent 工作流
- 独立向量服务
- 复杂 rerank pipeline
- 服务端永久聊天历史

## 演进原则

最重要的原则：

```text
先把它做成懂 gofurry 的搜索和解说层，而不是做成什么都能聊的 AI 助手。
```

具体取舍：

- 优先服务站内内容，不追求通用问答
- 优先稳定、可解释、可回滚，不追求模型链路复杂度
- 优先真实用户入口，不做孤立展示功能
- 优先一个人能维护的方案，不引入重基础设施
- 先让它找得准、找得稳、找得到依据，再让它回答得更像聊天产品
